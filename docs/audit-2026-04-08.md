## Detailed Findings

### Phase 1: Code Quality

#### 1.1 Architecture and Design

**Finding 1 (Low) -- Manual dependency wiring via Application casts**

All ViewModels and Workers obtain the database via `application as ProtocolApp`. This is a common Android pattern for small apps but becomes fragile as the app grows. Files affected:
- `TodayViewModel.kt:37`
- `PlanViewModel.kt:37-38`
- `WeekViewModel.kt:28-29`
- `PlanningReminderWorker.kt:20`

*Suggested fix*: Consider introducing Hilt or Koin for DI when the app grows beyond its current scope. Not urgent for a personal-use app.

**Finding 2 (Info) -- DataStore retained alongside SQLite**

The app migrated from DataStore to SQLite but still declares the DataStore dependency and keeps `ChecklistRepository` and `PrefsKeys` with DataStore-based methods. These are only used by the migration path.

*Suggested fix*: After the migration is considered stable, remove the DataStore dependency and consolidate repository code. See also Finding 7.

#### 1.2 Code Correctness

**Finding 3 (High) -- Race condition in migration startup**

`ProtocolApp.onCreate()` launches migration on `Dispatchers.IO` without any mechanism to block database access until migration completes. If a ViewModel queries the DB before migration finishes, it will see an empty/partial state.

File: `ProtocolApp.kt:25-27`

```kotlin
CoroutineScope(Dispatchers.IO).launch {
    DataStoreMigration.migrateIfNeeded(dataStore, db)
}
```

*Suggested fix*: Use a `CompletableDeferred` or `Mutex` that ViewModels await before first DB access, or run migration synchronously in `SQLiteOpenHelper.onOpen`.

**Finding 4 (Medium) -- WeekViewModel computes summary from a stale date list**

`WeekViewModel.init` captures `val days = (6 downTo 0).map { today.minusDays(it.toLong()) }` once at construction time. If the ViewModel survives past midnight, the "Last 7 Days" list becomes stale.

File: `WeekViewModel.kt:48-49`

*Suggested fix*: Recompute the day list when the schedule flow emits, or use a periodic refresh.

**Finding 5 (Medium) -- `scheduleMovementFollowUp` TOCTOU on `LocalTime.now()`**

`NotificationScheduler.scheduleMovementFollowUp` calls `LocalTime.now()` twice: once to check the time and once to compute the delay. Between the two calls, the time advances, making the delay slightly shorter than intended.

File: `NotificationScheduler.kt:113-128`

*Suggested fix*: Capture `now` once and reuse it.

#### 1.3 Code Hygiene

**Finding 6 (High) -- Dead code: `ChecklistRepository`**

`ChecklistRepository.kt` contains DataStore-based checklist methods that are no longer called by any ViewModel. After the SQLite migration, all checklist operations go through `ProtocolDatabase` directly.

Files: `ChecklistRepository.kt` (entire file), `PrefsKeys.kt` (methods `checklistKey`, `checklistTimesKey`, `gymCountKey`, `isoWeekId`)

*Suggested fix*: Delete `ChecklistRepository.kt`. Remove unused methods from `PrefsKeys.kt`. Keep only `SCHEDULE_JSON` (used by migration) until DataStore is fully removed.

**Finding 7 (Low) -- Unused colour constant**

`TealDark` in `Color.kt:8` is declared but never referenced.

*Suggested fix*: Remove it.

**Finding 8 (Low) -- Magic number for gym target**

The gym target of 3 sessions per week appears as a literal in:
- `TodayScreen.kt:171` (`"Gym this week: $gymCount / 3"`)
- `WeekScreen.kt:98` (`"$gymCount / 3 sessions"`)
- `WeekScreen.kt:103` (`gymCount / 3f`)

*Suggested fix*: Extract to a named constant, e.g., `const val GYM_WEEKLY_TARGET = 3`.

**Finding 9 (Info) -- Stale `BUILD` file at repo root**

A file named `BUILD` exists at the repo root. This appears to be either a stale Bazel build file or an artifact. It contains no meaningful content for the Gradle-based project.

*Suggested fix*: Verify and delete if unused.

### Phase 2: Security

**Finding 10 (High) -- `local.properties` committed to version control**

`local.properties` contains the local Android SDK path (`sdk.dir=/Users/marcelo/Library/Android/sdk`). While this specific value is not a secret, this file commonly contains signing keys and should never be checked in. It is a standard `.gitignore` entry for Android projects.

File: `local.properties`

*Suggested fix*: Add `local.properties` to `.gitignore` (it is already listed, but the file is tracked). Run `git rm --cached local.properties`.

**Finding 11 (Info) -- No secrets detected**

No hardcoded API keys, passwords, tokens, or private keys found in the codebase. The gather script's security scan confirmed zero matches.

### Phase 3: Testing

**Finding 12 (High) -- Zero test coverage**

The project has no tests whatsoever -- no unit tests, no instrumented tests, no test directories. This means:
- Data migration logic (`DataStoreMigration`) is untested
- Database schema upgrade (`onUpgrade`) is untested
- Business logic (checklist computation, schedule merging, parking planning) is untested
- Notification scheduling logic is untested

*Suggested fix*: Prioritise tests for:
1. `ProtocolDatabase` -- schema creation, CRUD operations, `onUpgrade` migration
2. `DataStoreMigration.migrateIfNeeded` -- migration correctness
3. `checklistFor()` -- returns correct items per day type
4. `PlanningRepository.unplannedDays` -- planning logic
5. `NotificationScheduler.delayUntil` -- time calculation edge cases

### Phase 4: Performance

**Finding 13 (Medium) -- WeekViewModel loads all 7 days synchronously in `init`**

`WeekViewModel.init` calls `db.completedItems(date)` 7 times sequentially inside `collect`. Each call is a separate SQLite query.

File: `WeekViewModel.kt:52-57`

*Suggested fix*: Consider a single SQL query with `WHERE date IN (?, ?, ...)` to fetch all 7 days at once.

### Phase 5: Build, CI/CD, and Infrastructure

#### 5.1 Build System

**Finding 14 (Medium) -- Missing ProGuard rules for kotlinx-serialization**

The release build enables minification (`isMinifyEnabled = true`) but there is no `proguard-rules.pro` file in the project. kotlinx-serialization requires specific keep rules to avoid stripping serialisers at build time.

File: `app/build.gradle.kts:27`

*Suggested fix*: Create `app/proguard-rules.pro` with:
```
-keepattributes *Annotation*, InnerClasses
-dontnote kotlinx.serialization.AnnotationsKt
-keepclassmembers class kotlinx.serialization.json.** { *** Companion; }
-keep,includedescriptorclasses class com.marcelo.protocol.**$$serializer { *; }
-keepclassmembers class com.marcelo.protocol.** {
    *** Companion;
}
-keepclasseswithmembers class com.marcelo.protocol.** {
    kotlinx.serialization.KSerializer serializer(...);
}
```
